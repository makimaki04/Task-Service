# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

## Swagger

Swagger добавлен для удобной ручной проверки API.

Swagger UI:

```text
http://localhost:8080/swagger/
```

OpenAPI JSON:

```text
http://localhost:8080/swagger/openapi.json
```

В Swagger описаны:

- обычные задачи
- шаблоны периодических задач
- схемы запросов и ответов для нового функционала

## Postman

Для ручного тестирования в репозиторий добавлена коллекция Postman:

```text
Postman/TaskTracker.postman_collection.json
```

Коллекция содержит минимальный набор запросов для тестирования:
- обычных задач
- создания шаблонов разных типов
- получения списка и одной записи шаблона
- обновления шаблона
- мягкого удаления шаблона
- просмотра сгенерированных задач

## API

Базовый префикс API:

```text
/api/v1
```

Основные маршруты:

- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `GET /api/v1/tasks/{id}`
- `PUT /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`
- `POST /api/v1/templates`
- `GET /api/v1/templates`
- `GET /api/v1/templates/{id}`
- `PUT /api/v1/templates/{id}`
- `DELETE /api/v1/templates/{id}`

## Что добавлено

Добавлена поддержка периодических задач через шаблоны задач.

Поддерживаются такие варианты периодичности:

- каждый `n`ый день
- каждое определенное число месяца от `1` до `30`(в случае 30, принято допущение о пропуске февраля)
- только на конкретные даты
- только на четные или нечетные дни месяца

Шаблон задачи хранит правило, по которому должны появляться задачи.
Сами задачи создаются как отдельные записи в трекере и дальше существуют по дефолтному циклу жизни со статусами `new`, `in_progress`, `done`.

## Принятые допущения

- Для периодических задач используется отдельная сущность шаблона. Это нужно, чтобы хранить правило отдельно от уже созданных задач.
- Обычные разовые задачи по-прежнему можно создавать напрямую через `/tasks`. Для них `template_id` не заполняется.
- Шаблон и реальные задачи разделены. Шаблон `template` отвечает за правило создания, а `tasks` отвечают за уже созданные рабочие задачи.
- При создании задачи из шаблона `template` в `tasks` копируются `title` и `description` на момент создания. Если потом шаблон изменится, уже созданные задачи не меняются.
- Поле `scheduled_for` в задаче означает дату, на которую создан конкретный экземпляр задачи.
- Если шаблон обновляется, это влияет только на будущую генерацию задач. Уже созданные задачи не изменяются и не удаляются.
- При обновлении шаблона система пытается создать недостающие будущие задачи по новым правилам. Если задача на такую дату уже существует, дублирующая запись не создается. Предыдщуие задачи не изменяются.
- Для шаблонов с `specific_dates` список дат при обновлении заменяется целиком.
- Если шаблон был типа `specific_dates`, а потом его поменяли на другой тип, старый список конкретных дат очищается.
- Удаление шаблона реализует мягкое удаление: запись не удаляется из базы, а переводится в деактивированный статус `active = false`. В будущем можно восстановить шаблон + реализовать ttl.
- Если в месяце такого числа нет, этот месяц просто пропускается.
- Для периодичности используются календарный дни, без уточнения времени суток.

## Про генерацию задач

В проекте нет фонового планировщика или отдельного воркера, который постоянно создает новые задачи по шаблонам.

Из-за этого генерация работает так:
- задачи создаются при создании шаблона
- задачи дополнительно генерируются при обновлении шаблона

Если у шаблона не задан `end_date`, задачи создаются не бесконечно, а только на ограниченный горизонт вперед.
В текущей реализации это ближайшие `30` дней. За данное значение отвечает переменная `defaultHorizonDays int = 30`, заданная в `template_service.go`

Это упрощение сделано специально, чтобы не добавлять в тестовое отдельный фоновый процесс.
Если развивать решение дальше, следующим шагом нужно добавить фоновый планировщик, который будет регулярно создавать новые задачи по активным шаблонам.
